# Руководство по стандартам, архитектуре и прагматизму Laravel (v10 - v13)

Этот документ — главный источник правды (Single Source of Truth) для проектирования, кодирования, рефакторинга и логирования изменений мультиагентной системой. ИИ-агент обязан строго соблюдать баланс между архитектурной чистотой (SOLID, DRY) и прагматизмом (KISS, YAGNI).

---

## 1. Главный инженерный баланс (Философия проекта)

При генерации кода и изменении архитектуры ИИ должен взвешивать решения по следующей шкале:
* **KISS (Keep It Simple, Stupid):** Пиши код так, чтобы его понял разработчик любого уровня. Избегай усложнений, если задача эффективно решается стандартными методами Laravel.
* **YAGNI (You Aren't Gonna Need It):** Не создавай классы, интерфейсы, DTO или Actions "на будущее". Только под текущую бизнес-задачу.
* **DRY (Don't Repeat Yourself):** Если логика дублируется более 2 раз — выноси её. Но помнить: небольшое дублирование лучше плохой абстракции.
* **SOLID:** Разделяй ответственность. Контроллер не знает о БД, Модель не знает об HTTP, Сервис/Экшен делает одну логическую задачу.

---

## 2. Архитектурный поток данных

Мы строго придерживаемся слабой связанности. Допустимая цепочка вызовов:
`Входящий запрос (FormRequest) -> Контроллер -> Сервис / Экшен -> Модель (Eloquent) -> API Resource / Blade`

### 🛡️ Валидация (Form Requests)
* **KISS/YAGNI:** Используй `FormRequest`, только если правил валидации > 5, требуется сложная кастомная логика или код будет переиспользован. Для простых полей (например, удаление или поиск) допускается инлайн-валидация в контроллере.
* Для статусов и ролей всегда применяй PHP Enums и правило `Rule::enum()`.

### 🎮 Контроллеры (Controllers)
* Делай контроллеры максимально тонкими (до 15-20 строк). 
* **Задача контроллера:** Принять запрос -> Вызвать бизнес-логику -> Вернуть ответ (Blade или Resource).
* Для сложных, объемных эндпоинтов используй контроллеры одного действия (`__invoke`).

### ⚙️ Бэкэнд: Разделение логики (Services vs Actions)
* **Простой CRUD:** Если действие тривиально (например, `User::create($request->validated())`), **ЗАПРЕЩЕНО** создавать пустые Сервисы-прокладки. Вызывай модель прямо из контроллера (Принцип YAGNI).
* **Классы-Сервисы (`app/Services`):** Используй, когда бизнес-логика сложная (>12 строк), затрагивает несколько моделей, транзакции или сторонние API (Принцип SOLID). Классы сервисов должны быть объявлены как `final`.
* **Классы-Действия (`app/Actions`):** Если Сервис раздувается и содержит разнородную логику, разбивай его на изолированные single-task классы: `CreateOrderAction`, `CancelOrderAction`. Они также должны быть `final`.

### 📦 Трансформация данных (DTO, Value Objects & spatie/laravel-data)
* Для API-ответов всегда используй `JsonResource`.
* **DTO & Value Objects:** Для передачи сложных, сильно вложенных или типизированных структур данных между сторонними API и Сервисами используй Data-классы из пакета `spatie/laravel-data`.
* Все DTO и Data-классы обязаны быть `final`. Для простых плоских массивов используй строгую типизацию PHP без создания лишних DTO (KISS).

## 3. Слой Eloquent, Наблюдатели, Скоупы и Трейты

* **Чистые модели:** Модели содержат только: отношения (relations), приведение типов (`casts`), аксессоры (современный синтаксис `Attribute::make()`) и локальные области видимости.
* **Локальные области видимости (Local Scopes):** Любое условие выборки, отражающее бизнес-статус (например, `where('status', Status::Published)`), должно быть вынесено в `scopePublished()`. Писать сырые `where` с бизнес-логикой в сервисах — запрещено (Нарушение DRY).
* **Наблюдатели (Observers):** Используй строго для побочных эффектов (логирование, очистка кэша, генерация слагов). Бизнес-логики там быть не должно.
* **Трейты (Traits):** Используй **только** для горизонтального разделения кода между *абсолютно несвязанными* моделями (например, `HasUuid`, `LogsActivity`). 
    * *Антипаттерн:* Запрещено резать толстую модель или контроллер на 5 трейтов (`UserRelations`, `UserAttributes`) просто ради уменьшения размера файла.

## 4. Стратегия тестирования (QA & Testing)

Каждая новая фича должна быть покрыта тестами. Мы используем **Pest** (предпочтительно) или **PHPUnit**.
* **Unit-тесты:** Для изолированных вычислений, хелперов, кастомных правил валидации и DTO. Без обращения к БД.
* **Feature-тесты:** Для проверки цепочки `HTTP Request -> Controller -> Service -> DB`. Тестируй конечные эндпоинты, проверяй статус ответа, структуру JSON или наличие данных в БД через `$this->assertDatabaseHas()`.
* **Тестирование Сервисов:** Тестируй класс Сервиса/Экшена напрямую, изолируя внешние API через Mocking (`Http::fake()`).
* **Browser-тесты (при необходимости):** Используй Laravel Dusk только для критических сквозных сценариев (End-to-End), где важна отработка JS/Vue на фронтенде.

## 5. Фронтенд: Blade, Vue, Стили и Локализация (i18n)

> ⚠️ **КРИТИЧЕСКОЕ ОГРАНИЧЕНИЕ:** Пакеты **Inertia.js, Livewire, Filament НЕ используются**. Генерировать код под них категорически запрещено.

* **Шаблонизатор:** Базовый рендеринг и роутинг — строго на **Blade** + серверные компоненты макетов (`<x-layout>`).
* **Реактивность:** Vue.js используется **точечно**, в виде изолированных виджетов внутри Blade-страниц. Данные во Vue передаются через пропсы из Blade или через стандартные API-запросы.
* **Стили:** 95% верстки — утилитарные классы **Tailwind CSS**. **SCSS** разрешен только для сложных анимаций (`@keyframes`) или кастомизации тяжелых сторонних библиотек. Не дублируй Tailwind в `@apply`.
* **Перспектива i18n:** Все интерфейсные строки (как в Blade, так и в Vue/JS) должны выводиться через языковые файлы: `__('messages.welcome')` или `{{ $t('welcome') }}`. Хардкодить текст прямо в разметку запрещено.

## 6. Интеграция с Laravel Boost (MCP)

Проект использует `laravel/boost` как MCP-сервер для ИИ.
- Ты обязан использовать MCP-сервер Laravel Boost для проверки структуры базы данных, вызова безопасных Eloquent-запросов через Tinker и запуска тестов (Pest/PHPUnit).
* **Ты обязан:** Инспектировать реальную схему БД через инструменты Boost, чтобы не выдумывать имена колонок и индексов.
- **Критическое правило:** Инструменты Boost используются исключительно для *анализа, верификации и тестирования* текущего состояния приложения. 
- Запрещено использовать генераторы Boost, если они нарушают зафиксированный поток данных: FormRequest → Controller → Service → Model → Blade + Vue.
