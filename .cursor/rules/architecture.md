# Архитектура и Поток Данных

## Допустимый поток данных:
`FormRequest (Валидация) → Controller (Маршрутизация) → Service/Action (Бизнес-логика) → Model (Eloquent) → Resource/Blade (Интерфейс)`

## Матрица слоев архитектуры (Защита от оверинжиниринга)

| Архитектурный слой | Когда Использовать (SOLID/DRY) | Когда НЕ использовать (KISS/YAGNI) |
| :--- | :--- | :--- |
| **FormRequest** | Правил валидации > 5, сложная или повторная валидация. | Простые формы (2-3 поля), тривиальный поиск/удаление. |
| **Service (Класс-Сервис)** | `final` класс. Бизнес-логика > 12 строк, транзакции, интеграция внешних API. | Простые CRUD-операции (`Model::create()`, `Model::update()`). |
| **Action (Класс-Действие)** | `final` класс одного действия. Когда Сервис нарушает Single Responsibility. | Когда логика укладывается в 1-2 простых метода Сервиса. |
| **DTO (spatie/laravel-data)**| `final` класс. Передача сложных, вложенных структур данных из API. | Плоские простые массивы (используй типизацию PHP). |
| **Local Scope (Eloquent)** | Любые повторные выборки по бизнес-критериям (статусы, даты). | Одноразовые, специфические системные запросы. |
| **Repository (Репозиторий)** | **ЗАПРЕЩЕНО**. Заменено на Eloquent + Local Scopes. | В 100% случаев в проекте. |

*Принцип:* Начинай с самого простого (KISS). Усложняй и добавляй слои только тогда, когда код начинает дублироваться или становиться "грязным".
